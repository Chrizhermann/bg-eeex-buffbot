# EEex v0.11 Compatibility — Design Document

**Date:** 2026-07-20
**Status:** Approved
**Target:** BuffBot v1.6.1-alpha

## Problem

BuffBot v1.4.0 and newer currently refuse to install on EEex v0.11.0-alpha. The
installer treats the v1-only `EEex_scripts/EEex_Sprite.lua` layout as a version
requirement, even though the production APIs used by BuffBot v1.6.0-alpha are
also present in official EEex v0.11.0-alpha.

The rejection is therefore mostly artificial. There are, however, two real
runtime differences that must be handled before advertising v0.11 support:

1. EEex v0.11 marshal handlers accept only numbers, strings, and tables.
   A nested boolean or unsupported value can make saving fail or crash.
2. EEex v1 contains listener-error isolation that v0.11 lacks. An uncaught
   error in a BuffBot callback can propagate through the surrounding EEex
   event path on v0.11.

## Goals

- Support EEex v0.11.0-alpha and EEex v1.0.0+ with the same BuffBot feature set.
- Reject EEex v0.10 and incomplete/stale EEex installations clearly.
- Avoid depending on EEex component numbers, which changed between releases.
- Guarantee that BuffBot's marshal export contains only v0.11-safe values.
- Contain and log errors at BuffBot-owned EEex callback boundaries.
- Prove installer behavior without modifying a live game installation.
- Keep EEex v1 as the recommended version.

## Non-Goals

- Supporting EEex v0.10 or older.
- A separate legacy BuffBot branch or reduced-feature compatibility mode.
- Changing summon detection, summon execution, Project Image handling, or
  copied F12 innate behavior.
- Promising that an arbitrary save already written by EEex v1 and other mods
  can be downgraded safely to v0.11.
- Investigating the reported new-EEex/CDTweaks/Rogue Rebalancing large-install
  failure. That reproduction remains a separate follow-up after this patch.

## Considered Approaches

### 1. Capability- and layout-based compatibility (chosen)

Recognize the official v0.11 and v1 script layouts, validate the modules
BuffBot actually requires, and harden the marshal and callback boundaries.
This keeps one runtime behavior path and avoids coupling BuffBot to EEex's
changing component numbers.

### 2. Parse an EEex version string or WeiDU component IDs

Rejected. EEex main moved from component 0 to component 1, LuaJIT moved from
component 1 to component 8, and older releases do not expose one stable version
contract. Version-text parsing would also encode installer formatting rather
than the capabilities BuffBot needs.

### 3. Maintain a legacy package or compatibility mode

Rejected. The required runtime APIs already exist in v0.11, so a second code
path would duplicate testing and eventually drift without providing a user
benefit.

## Design

### Installer prerequisite

Both BuffBot components will use the same version-independent prerequisite:

```text
EEex bootstrap exists
AND
(
    old-layout Sprite and Options modules exist and Sprite has the v0.11 API
    OR
    new-layout Sprite and Options modules exist and Sprite has the v0.11 API
)
```

Concretely:

- `override/M___EEex.lua` is the installed bootstrap marker.
- EEex v0.11 supplies `EEex_Sprite.lua` and `EEex_Options.lua` in `override/`.
- EEex v1 supplies those modules in `EEex_scripts/`.
- The selected `EEex_Sprite.lua` must contain the required
  `EEex_Sprite_GetCharacterIndex` capability. This prevents a downgraded or
  stale installation from passing merely because an old Options file remains.
- Requiring both modules and the capability excludes v0.10, which does not
  provide the complete API surface BuffBot needs.

The check will not use `MOD_IS_INSTALLED` component IDs. The main and LuaJIT
components will use the same predicate text, with a static regression test that
prevents their requirements from drifting apart.

The existing LuaJIT setup remains one code path. It already distinguishes the
old `LuaVersionExternal=5.1` value from v1's `5.1-LuaJIT` value by inspecting
the shipped EEex installer. "Active" will mean that `InfinityLoader.ini` has
the external patch mode, `LuaLibrary=lua51.dll`, an accepted external-version
value, and both root DLLs exist. An incomplete state is repaired or fails
cleanly rather than being accepted from the patch-mode line alone.

### Marshal-safe persistence

BuffBot will add a recursive marshal-safe copy operation at the `_Export`
boundary. It will:

- Preserve numbers and strings.
- Convert booleans to integer `1` or `0`.
- Recursively copy supported table keys and values. Only number and string
  keys are accepted.
- Drop functions, userdata, threads, and other unsupported values with a
  diagnostic warning.
- Drop boolean and unsupported table keys instead of converting them; key
  conversion could collide with an existing numeric key (`true` and `1`).
- Detect cycles and drop the cyclic branch rather than recursing forever or
  risking a failed save.
- Leave the live UDAux configuration untouched.

Imported and externally loaded configurations will continue through schema
validation. The export boundary remains the final invariant: even a malformed
value introduced mid-session cannot reach the v0.11 marshaler.

The sanitizer is not a schema migration and does not increment schema v8.

### Listener error containment

The core module will expose one small callback wrapper that:

- Invokes the BuffBot callback through `pcall`.
- Preserves successful return values where the EEex callback contract uses
  them.
- Reports the callback label and error once through `BfBot._Error`, making it
  visible and writing it to the active log when one is open.
- Prevents the error from propagating into EEex v0.11.
- Suppresses repeated identical warnings from high-frequency callbacks.

It will be applied to BuffBot-owned event entry points where an exception can
escape into EEex: main-menu load, sprite load/late join, innate refresh, key and
spell-list notifications, and window-size changes. Existing render hooks that
already contain their own guards will not be broadened unless a failing test
demonstrates a need.

The existing silent inner `pcall` around innate refresh will be replaced, not
merely wrapped from the outside. The no-LuaJIT notification listener will be
registered only after the core helper has loaded, avoiding a bootstrap-order
exception to the policy.

### Documentation and versioning

- README requirements become EEex v0.11.0-alpha or newer.
- EEex v1 remains recommended because it protects its scripts from generalized
  biffing and has safer native marshal/listener behavior.
- The README will caution that later mods must not biff or overwrite v0.11's
  EEex Lua files in `override/`.
- The changelog will explain why the previous v1-only floor was conservative.
- Code and installer versions will be prepared as v1.6.1-alpha. Creating a tag
  or GitHub release remains a separate explicit release action.

## Verification Strategy

### Automated installer tests

A Python integration test will create a disposable synthetic BG2EE game with a
minimal KEY/BIFF/TLK and run the public `setup-buffbot.tp2`. It will cover:

- No EEex: both components are skipped with the v0.11 requirement message.
- WeiDU.log entries without runtime files: rejected.
- v0.10-like old layout missing the Options module: rejected.
- v0.10-like layout with a stale Options module but without
  `EEex_Sprite_GetCharacterIndex`: rejected.
- Complete v0.11 old layout: accepted.
- Complete v1 new layout: accepted.
- Partial/stale layout without the bootstrap: rejected.
- Component-number-independent acceptance, including fixtures without an
  EEex WeiDU.log entry.
- v0.11 and v1 LuaJIT values, already-active behavior, incomplete-state repair,
  controlled failure, and byte-exact rollback.

The core matrix will run with WeiDU 249 and 251 when both binaries are locally
available. Tests will compare the meaningful game payload and restoration
state rather than treating WeiDU's own logs/backups as product output.

### Runtime tests

Test-first Lua coverage will prove:

- Deeply nested booleans become `1`/`0` in exported data.
- Supported values survive unchanged.
- Unsupported values and cycles cannot escape the export boundary.
- Boolean/unsupported keys are dropped without colliding with numeric keys.
- Sanitization does not mutate the live configuration.
- A wrapped callback returns normally on success and logs instead of
  propagating an error on failure.

### Live acceptance

After automated checks pass, use disposable v0.11 and v1 game copies. Do not
modify or launch against the active large installation. On each version verify:

- Full `BfBot.Test.RunAll()`.
- Panel/options opening and configuration.
- Save, reload, and save again.
- Normal and Quick Cast.
- F12 reconciliation.
- Summon discovery, standalone and joint casting, late join, expiry, and the
  Project Image owner-lock path.

The separate large-install/new-EEex reproduction begins only after this
compatibility patch is complete.

## Risks and Mitigations

- **False-positive EEex detection:** require the bootstrap and both required
  modules in one recognized layout; test partial/stale fixtures.
- **Save corruption from unexpected values:** enforce a non-mutating safe copy
  immediately before marshal export and exercise hostile nested fixtures.
- **Silent callback failure:** log contained exceptions with a stable callback
  label instead of using unchecked or silent `pcall` calls.
- **Live-install damage:** run WeiDU tests only in generated fake games and use
  disposable copies for runtime validation.
- **Scope creep into the reported mod-stack problem:** track it separately and
  do not install or diagnose the large stack during this patch.
